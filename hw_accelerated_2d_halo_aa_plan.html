HW-Accelerated 2D Halo AA — Future Plan

FUTURE Documented 2026-05-25. Not currently scheduled. Discussed after landing the agg-sharp texture-AA port (commit 1df4311).

Status: Validation document. The plan below has been walked through but is not implemented. Today the aa_texture_pipeline tessellates the AGG-Sharp-style mesh on the CPU every frame, in screen space, after the CTM is applied. That is correct and renders pixel-identical to agg-sharp; it just re-runs the work each frame. This document is the design for moving that work into the vertex shader so static paths can tessellate once and re-render at any zoom, rotation, or 3D orientation.

Where we are today

agg-gui/src/gl_renderer/aa_texture_mesh.rs ports AARenderTesselator.cs from agg-sharp:

  • Per AA edge, emit a 5-vertex triangle fan with three triangles (extruded strip × 2 + interior).
  • U-coordinates are computed in screen space: 1/1023 at the polygon edge, 0 at the 1-pixel-outward extruded vertex, (1 + edgeDotP3)/1023 at the non-AA interior point.
  • Sampled against a 1024×1 alpha-step texture (col 0 α=0, cols 1+ α=opaque) with LINEAR filtering.
  • Premultiplied blend (One / OneMinusSrcAlpha).

The mesh is built fresh inside WgpuGfxCtx::do_fill and do_stroke after applying the CTM via ConvTransform, so zoom / rotation / pan all force a re-tessellation. Lion's CachedTess + expand_aa_halo pattern caches the interior triangles' edge-flag data but still rebuilds the halo geometry per frame.

The premise

Move the screen-space half of the work into the vertex shader so the mesh can live in path space.

Every transform-dependent quantity AGG-Sharp computes at tess time decomposes cleanly into two parts:

  1. Vertex position transform — commutes with CTM application order; can be deferred.
  2. 1-screen-pixel perpendicular extrusion — a screen-space operation; easy to compute in the VS given the CTM and a per-vertex flag.

With both deferred, the mesh becomes constant for a path's lifetime and the CTM rides as a uniform.

Vertex format

FieldTypeBytesMeaning
posvec28Path-space anchor.
edge_dirvec28Path-space edge direction, signed so that perp(edge_dir) already points outward (CPU pre-flips at tess time).
u_basef324Screen-pixel-constant numerator of U. 0 for extruded vertex, 1 for polygon edge, 0.2 × 1023 / 0.9 × 1023 for DrawNonAATriangle.
u_path_partf324Path-space distance contribution to U (the edgeDotP3 for the non-AA interior point; 0 otherwise). Scaled to screen pixels by the VS.
extrudef3240 = vertex stays at the anchor; 1 = displaced 1 screen pixel along perp(edge_dir).

Total 28 bytes per vertex (1.75× the current 16-byte AaTexVertex). v coordinate is dropped because the 1024×1 step texture is V-invariant; add it back as a 32-byte format if textual parity with agg-sharp matters.

Vertex shader sketch

let anchor_screen = apply_ctm(in.pos);
let edge_screen   = apply_ctm_linear(in.edge_dir);
let edge_len      = length(edge_screen);
let edge_unit     = select(vec2<f32>(1.0, 0.0),
                           edge_screen / edge_len,
                           edge_len > 1e-6);
let perp_screen   = vec2<f32>(edge_unit.y, -edge_unit.x);

// 1-screen-pixel halo extrusion, always in screen space.
let final_pos = anchor_screen + in.extrude * u.halo_px * perp_screen;

// U coord: constant part + path-space part scaled to screen pixels.
// For pure rotation, perp_scale = 1; for 2x zoom, 2; etc.
let perp_scale = edge_len;
let u_screen   = in.u_base + in.u_path_part * perp_scale;
let uv         = vec2<f32>(u_screen / 1023.0, 0.5);

return VOut(ndc(final_pos), uv);

Fragment shader is unchanged from the current AA_TEXTURE_WGSL: sample the alpha-step texture, multiply by uniform color, output premultiplied.

Validation by case

Pure translation / uniform zoom / 2D rotation (the common case)

  • edge_screen is edge_dir rotated and uniformly scaled. edge_len = |scale|.
  • perp_screen is the correct screen-space outward perpendicular.
  • final_pos extrudes by exactly halo_px screen pixels along that perpendicular.
  • u_screen = u_base + path_edgeDotP3 × scale = u_base + screen_edgeDotP3 — bitwise-identical to what the current CPU tessellator computes.

Non-uniform scale (e.g. 2× horizontal, 1× vertical)

  • Horizontal edge (path normal = vertical): edge_screen = (2, 0), edge_len = 2, perp_screen = (0, -1), halo extends 1 screen px vertically.
  • Vertical edge (path normal = horizontal): edge_screen = (0, 1), edge_len = 1, perpendicular extends 1 screen px horizontally.
  • U scaling tracks the perpendicular component correctly because perp_screen is recomputed from the rotated/scaled edge in screen space.

Shear

  • Path-space edge (1, 0) becomes screen-space (1, k) under shear matrix [[1,0],[k,1]].
  • edge_unit = (1, k) / sqrt(1 + k²). perp_screen is the screen-space perpendicular — correct.
  • Halo extends 1 screen pixel along the screen perpendicular, not the path perpendicular. Correct under shear, which the current CPU port can't claim (it would extrude along the rotated path normal, which is no longer perpendicular to the screen edge under shear).

3D rotation of the 2D plane (orthographic)

  • The linear part of the camera matrix transforms the in-plane edge direction into screen space.
  • edge_unit and perp_screen are computed in screen space directly.
  • Halo width stays 1 screen pixel regardless of viewing angle.
  • The polygon foreshortens (less in-plane edge per screen pixel) and edge_len drops accordingly — which is what we want: the halo is in screen pixels, the polygon is in projected pixels, the AA edge sub-pixel transition stays sub-pixel.

Perspective

  • Works to a first approximation — the on-screen edge length varies along the edge under perspective. Using the endpoint edge_dir gives the average.
  • For extreme foreshortening you'd want per-fragment perpendicular reconstruction in the FS. Out of scope for typical 2D UI; flag if it ever matters.

Sub-pixel-wide polygon at any zoom

  • path_edgeDotP3 is small in path space; screen_edgeDotP3 = path_edgeDotP3 × scale.
  • If the polygon is < 1 screen pixel wide, screen_edgeDotP3 < 1 and u_screen lands inside the texel-0-to-texel-1 LINEAR ramp — sub-pixel coverage matches agg-sharp exactly.

Adjacent shapes (the pixel test)

  • Each shape's extruded vertices land 1 screen pixel outside that shape's edge.
  • The neighbour's pixel centres still sample the alpha-step texture at U ≈ 0 (texel 0, α = 0).
  • No bleed; pixel test stays clean.

Gotchas to flag

1. Edge-direction sign must be canonical. AGG-Sharp's perpendicular sign flip (Vector2.Dot(normal, nonAaPoint - aaEdgeP0)) must move into the mesh: the CPU encodes edge_dir such that perp(edge_dir) already points outward. The VS doesn't need to know about nonAaPoint.
2. Caching is half the win, and the cache key is the hard part. The benefit is "tessellate once, render at any CTM" — but only if the path geometry is stable across frames.
  • Who decides what's cacheable? Either we hash every ctx.rect / line_to sequence inside WgpuGfxCtx, or we add a deliberate "this path is reusable" API the way tessellate_interior + expand_aa_halo already does for Lion. The natural fit is to extend CachedTess to also store an AaTexVertexVs mesh.
  • Animated paths (a Marbles ball whose path changes every frame) get zero benefit and pay the larger vertex stride. Provide a fallback: dynamic paths keep using today's screen-space CPU tessellation; static / cached paths use the VS-driven one. Same DrawCommand variant, different tessellator at the caller boundary.
3. Vertex size grows 16 → 28 bytes (1.75×). Trivial on desktop, fine on WebGL2, but on mobile WebGL2 vertex bandwidth can become a hot spot. Worth flagging — not blocking.
4. apply_ctm_linear precision under heavy zoom. At zoom factors > ~1e4 the f32 in edge_screen starts to lose precision before the normalise. Real-world UI never approaches that, but anyone using this for a CAD viewer should know. If it ever bites, keep the translation part of the CTM in f64 on the CPU side (matches agg-rust::TransAffine's precision split).
5. AGG-Sharp's per-endpoint unitsPerPixel falls out for free. In agg-sharp the CPU looks up world.GetWorldUnitsPerScreenPixelAtPosition at each endpoint of the AA edge, so a foreshortened 3D edge gets per-endpoint correct halo width. Our VS version does the same naturally — edge_len is per-vertex, perp_screen is per-vertex, vertex interpolation gives smooth per-fragment halo width along a foreshortened edge.
6. u_base for DrawNonAATriangle is a literal. Just storing 0.2 × 1023 (and 0.9 × 1023) as a vertex attribute keeps the VS uniform. With u_path_part = 0 for non-AA tris, u_screen = u_base, which divided by 1023 gives the correct 0.2 / 0.9 sample.

Bottom line

The premise is sound. The math holds for affine, shear, and orthographic 3D rotation. Sub-pixel coverage and the gray-stripes fix from the texture-AA approach we just landed both carry over unchanged. The agg-sharp-fidelity output we have today is preserved; what changes is that the CPU re-tessellation cost goes away for static paths.

The largest design call to make before coding is the caching surface: hash paths inside WgpuGfxCtx, or only optimise the explicit CachedTess-style API? That decision doesn't affect the VS port itself; it only affects which callers benefit and by how much.

When the time comes to do this, start at Phase 1 with the new tessellator and the CPU-reference unit test. The risk gates open cleanly from there.
